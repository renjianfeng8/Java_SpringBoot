## 1. 改造目标

本次改造的目标不是继续堆叠 CRUD 功能，而是把现有影院购票管理系统升级为可以用于简历投递、在线演示和面试深挖的工程化作品。

最终项目应能够体现：

1. Java 后端工程能力：权限、事务、订单一致性、参数校验、异常处理、数据库建模、测试。
2. 全栈交付能力：Vue 3 前端、REST API、CI/CD、Docker 化部署、线上演示。
3. 项目表达能力：README、架构图、业务流程图、简历 bullet、面试问答材料。
4. 可验证质量：核心流程可跑通，GitHub Actions 通过，关键业务有测试覆盖。

简历中的项目名称建议使用：

**多角色影院票务运营平台**

简历一句话描述：

基于 Spring Boot 3、Vue 3 和 MySQL 构建的多角色影院票务运营平台，支持用户购票、影院排片、平台审核、订单流转、权限隔离、CI/E2E 自动化验证与 Docker 化部署。

## 5. 总体架构设计

### 5.1 逻辑架构

```text
用户端 Vue 3
  - 首页/影片/影院/详情/选座/订单/评价

影院端 Vue 3
  - 影院资料/影厅/排片/订单

管理端 Vue 3
  - 用户/影院/影片/分类/地区/演员/公告/审核/运营数据

        |
        | Axios + JWT
        v

Spring Boot API
  - AuthController
  - Film/Cinema/Room/Record/Ordered/Mark Controllers
  - BaseController 通用 CRUD
  - AuthInterceptor RBAC 权限校验
  - GlobalExceptionHandler 统一异常
  - DTO/VO 参数和响应边界

        |
        | MyBatis Mapper
        v

MySQL
  - 用户、影院、影片、影厅、排片、订单、评价
  - 关键索引
  - 初始化数据
```

### 5.2 后端边界

后端保持单体架构，重点体现业务边界清晰，而不是微服务拆分。

- Controller：只负责参数接收、认证上下文获取、返回统一响应。
- Service：承载业务规则、事务、权限边界和状态流转。
- Mapper：只负责 SQL，不写业务规则。
- DTO/VO：隔离外部 API 和内部 Entity。
- Interceptor：处理认证和通用角色边界。
- ExceptionHandler：统一错误码、错误消息和 HTTP 语义。

### 5.3 前端边界

前端不做大规模重构，重点保证演示体验和工程一致性。

- `utils/request.js`：统一 token 注入和错误处理。
- `utils/authStorage.js`：统一登录态读取和清理。
- 路由守卫：未登录跳转、角色越权跳转。
- 重点页面：用户首页、电影详情、选座、订单、后台表格。

## 6. 后端改造设计

### 6.1 DTO/VO 策略

优先改造面试最容易追问的接口：

- 登录注册。
- 修改密码。
- 用户资料。
- 影院资料。
- 排片创建和更新。
- 订单创建、取消、取票。

设计原则：

- 请求参数使用 `Request` DTO。
- 响应对象使用 `VO`，不返回密码、内部标识、无关字段。
- 通用 CRUD 资源可逐步过渡，不要求一次性全量 DTO 化。
- 关键业务接口必须使用 DTO，避免 Entity 直接暴露。

### 6.2 参数校验

使用 Bean Validation：

- 登录：用户名、密码、角色不能为空。
- 注册：账号、密码、手机号或影院基础信息校验。
- 排片：影片、影厅、时间、价格不能为空，价格必须大于 0。
- 订单：排片 ID、座位信息、用户身份不能为空。
- 文件上传：文件不能为空，大小、后缀、MIME 类型必须符合白名单。

### 6.3 统一异常

定义业务异常和错误码：

| 错误类型 | 场景 |
|---|---|
| UNAUTHORIZED | 未登录或 token 失效 |
| FORBIDDEN | 角色无权限 |
| PARAM_INVALID | 参数校验失败 |
| NOT_FOUND | 资源不存在 |
| BUSINESS_CONFLICT | 重复购票、状态不允许 |
| FILE_INVALID | 上传文件非法 |
| SYSTEM_ERROR | 未预期异常 |

所有接口保持统一响应结构，便于前端统一处理。

### 6.4 权限和数据隔离

权限目标：

- ADMIN 可以管理全局资源。
- CINEMA 只能管理本影院相关影厅、排片、订单和资料。
- USER 只能查看和操作自己的订单、评价和个人信息。

需要产出：

- `docs/design` 或 `docs/interview` 中的 RBAC 权限矩阵。
- 后端关键接口的数据隔离校验。
- E2E 或后端测试覆盖至少一个越权场景。

### 6.5 订单一致性

订单链路需要形成可讲闭环：

1. 用户选择排片。
2. 查询已售座位。
3. 用户提交座位。
4. 后端校验排片存在、座位合法、座位未售出。
5. 后端在事务内创建订单并写入座位信息。
6. 重复座位下单返回业务冲突。
7. 用户可取消待取票订单。
8. 影院端或管理端可完成取票状态变更。

轻量实现优先级：

- 必须实现业务校验和事务边界。
- 建议增加数据库唯一约束或等价防护，防止并发重复购票。
- 可选增加订单超时取消定时任务。
- Redis 锁座作为后续增强，不阻塞本期上线。

## 8. 数据库设计

### 8.1 核心关系

核心关系必须在文档中讲清楚：

- `cinema -> room`
- `film -> record`
- `room -> record`
- `record -> ordered`
- `user -> ordered`
- `film -> mark`

### 8.2 索引策略

建议增加或确认以下索引：

- `room.cinema_id`
- `record.film_id`
- `record.room_id`
- `ordered.user_id`
- `ordered.record_id`
- `mark.film_id`
- 常用搜索字段：影片名称、年份、地区、类型关联字段。

### 8.3 初始化数据

演示数据要求：

- 至少 8 部电影。
- 至少 3 家影院。
- 每家影院至少 2 个影厅。
- 至少 10 条排片。
- 至少 5 条订单。
- 覆盖待取票、已取票、已取消状态。

## 10. 轻量化上线设计

### 10.1 推荐部署方式

推荐使用：

**云服务器 + Docker Compose**

理由：

- 更接近真实生产部署。
- 可同时部署 MySQL、后端、前端/nginx。
- 面试中更容易讲清楚环境变量、日志、健康检查和端口暴露。

备选：

- 前端 Vercel/Netlify + 后端 Render/Railway + 云 MySQL。

### 10.2 生产配置

需要新增或完善：

- `.env.example`
- `application-prod.yml`
- Docker Compose 环境变量。
- 数据库密码、JWT 密钥、上传目录、日志级别通过环境变量注入。
- 生产环境关闭 DEBUG SQL 日志。

### 10.3 健康检查

后端提供健康检查：

- 可使用 Spring Boot Actuator `/actuator/health`。
- 或提供轻量 `/api/v1/health`。

Docker Compose 和部署文档需要说明健康检查方式。

## 14. 风险与控制

| 风险 | 控制方式 |
|---|---|
| 范围膨胀 | 不做微服务、消息队列、Kubernetes 等超出本期目标的技术 |
| 上线环境耗时 | 优先 Docker Compose 单机部署，云平台部署作为备选 |
| 订单并发复杂度过高 | 先用事务和数据库约束完成可靠防护，Redis 作为可选增强 |
| DTO 全量改造过大 | 优先改造认证、订单、排片、影院等关键接口 |
| 前端重构影响稳定性 | 只打磨重点演示页面和通用错误处理 |
| 测试时间不足 | 先覆盖认证、权限、订单三类高价值用例 |
