# 术语表 Glossary

本文档统一定义项目中使用的角色、实体、业务流程和技术术语。所有产品需求文档引用术语时指向此处。

---

## 1. 角色与权限

| 术语 | 英文标识 | 定义 | 相关文档 |
|------|----------|------|----------|
| 系统管理员 | ADMIN | 平台全局管理者，可访问 `/manage/*` 全部 16 个管理页面，管理基础数据、审核影院、查看所有订单和评价 | [PRD §4](./03-product-requirements-document.md#4-角色与权限) |
| 影院管理员 | CINEMA | 影院运营者，可访问 `/back/*` 7 个影院后台页面，管理本影院资料、影厅、排片和订单 | [PRD §4](./03-product-requirements-document.md#4-角色与权限) |
| 普通用户 | USER | 观影消费者，可访问 `/front/*` 12 个用户端页面，浏览、购票、评价 | [PRD §4](./03-product-requirements-document.md#4-角色与权限) |
| 未登录用户 | GUEST | 未认证访客，仅可访问登录/注册页，受限接口返回 401 | [PRD §11.1](./03-product-requirements-document.md#111-功能验收) |
| 三端分离 | — | 系统按 ADMIN/CINEMA/USER 三种角色分离前端入口和后端权限的设计模式 | [用户需求 §5](./01-user-and-business-requirements.md#5-权限需求) |
| 路由守卫 | Route Guard | Vue Router 的导航守卫，根据角色拦截非法页面跳转 | — |
| AuthInterceptor | — | 后端 Spring Boot 拦截器，校验 JWT Token 并判断角色是否有权访问接口 | — |

## 2. 核心业务实体

### 2.1 账号类

| 术语 | 对应表 | 定义 |
|------|--------|------|
| 管理员账号 | `admin` | 系统管理员登录账号，包含用户名、密码、姓名、电话、邮箱、头像 |
| 用户账号 | `user` | 普通用户登录账号，字段同上 + 角色固定为 USER |
| 影院账号 | `cinema` | 影院管理员登录账号，额外包含地址、负责人、资质证书、联系电话、邮箱、简介和审核状态 |

### 2.2 内容类

| 术语 | 对应表 | 定义 |
|------|--------|------|
| 电影/影片 | `film` | 核心内容实体，包含标题、英文名、海报、片长、简介、制作方、预告链接、票房、上映日期、关联地区与分类 |
| 分类 | `type` | 电影类型标签，如"动作"、"喜剧"、"科幻" |
| 地区 | `area` | 电影制片地区，如"中国大陆"、"美国" |
| 演员 | `actor` | 参演电影的演员，含姓名和头像 |
| 预告片 | `video` | 电影预告视频，含标题、视频名称、预览 URL、封面图和上映日期 |
| 公告 | `notice` | 平台发布的公告信息，含标题、内容和发布时间 |

### 2.3 业务类

| 术语 | 对应表 | 定义 |
|------|--------|------|
| 影厅 | `room` | 影院内的放映厅，包含名称（如"1号厅"）和所属影院；当前用户端选座图为固定 8×8 |
| 排片/场次 | `record` | 一场电影放映安排，关联影院+影厅+电影+时间+价格+状态 |
| 订单 | `ordered` | 用户购票记录，关联用户+场次+电影+影院+影厅+座位+金额+状态 |
| 评价 | `mark` | 用户对电影的评分记录，当前以 `mark` 字段存储评分/评价内容 |
| 影院-电影关联 | `cinema_film` | 影院和电影的多对多关联，表示某影院可放映某电影 |
| 电影-分类关联 | `film_type` | 电影和分类的多对多关联，一部电影可有多个分类标签 |

## 3. 业务流程术语

| 术语 | 定义 | 涉及角色 |
|------|------|----------|
| 排片 | 影院管理员为某电影在指定影厅安排放映场次（时间+价格）的操作 | CINEMA |
| 场次 | 一个具体的放映安排，包含电影、影厅、开始时间、结束时间、价格 | USER, CINEMA |
| 选座 | 用户在 8×8 座位图上选择可售座位并提交订单的过程 | USER |
| 在线选座 | 用户远程查看座位图并选择座位的购票方式，当前为固定 8×8 网格 | USER |
| 核销 | 影院管理员确认用户已取票的操作，订单变为"已取票"状态 | CINEMA |
| 海报 | 电影宣传封面图，通过文件上传接口存储 | ADMIN |
| 票房 | 电影的累计售票收入，用于排行榜排序 | — |
| 评分 | 用户对电影的 1-5 星评价，平均后展示在详情页和排行榜 | USER, — |
| 审核 | 管理员对影院注册资料进行通过/驳回的操作 | ADMIN |

## 4. 订单状态

| 状态值 | 含义 | 说明 |
|--------|------|------|
| 待取票 | 订单已生成，等待用户到影院取票 | V1.0 下单后默认状态 |
| 已取票 | 用户已完成取票 | 后台将订单状态更新后进入 |
| 已取消 | 订单被用户取消或系统自动取消 | — |

> V1.0 仅包含上述三种状态。扩展状态（待支付、已支付、退款中、已退款）规划见 [PRD 优化路线图 §3.1](./04-prd-optimization-roadmap.md#31-订单状态机)

## 5. 技术术语

| 术语 | 全称 | 定义 |
|------|------|------|
| JWT | JSON Web Token | 基于 Token 的认证方式，登录后后端签发 Token，前端在后续请求中携带以证明身份 |
| RBAC | Role-Based Access Control | 基于角色的访问控制，通过 ADMIN/CINEMA/USER 三种角色控制资源访问权限 |
| CRUD | Create Read Update Delete | 增删改查四种基本数据操作 |
| RESTful | Representational State Transfer | 一种 API 设计风格，本系统使用 `/api/v1/{resources}` 路径模式 |
| BaseController | — | 后端通用 CRUD Controller 基类，提供 7 个标准 RESTful 端点 |
| BaseService | — | 后端通用 CRUD Service 基类，封装 CRUD 方法 + 事务管理 |
| BaseMapper | — | 后端通用 MyBatis Mapper 基类，定义 CRUD 方法 |
| E2E | End-to-End | 端到端测试，本系统使用 Playwright 模拟用户操作验证全链路 |
| MIME | Multipurpose Internet Mail Extensions | 文件类型标识，上传模块通过 MIME 白名单防止危险文件上传 |
| PageHelper | — | MyBatis 分页插件，通过 ThreadLocal 实现物理分页 |
