# PRD 优化路线图

## 1. 优化原则

基于当前项目现状（详见 [PRD](./03-product-requirements-document.md) 和[用户需求](./01-user-and-business-requirements.md)），后续 PRD 优化不建议盲目堆功能，而应围绕三条主线：

- **购票链路更真实**：支付、锁座、订单状态、取票码。（当前订单状态见 [PRD §8.1](./03-product-requirements-document.md#81-订单状态定义)）
- **影院运营更专业**：座位模板、票价策略、场次收入、上座率。
- **平台治理更安全**：对象级权限、审核流程、数据报表。

> 术语说明：业务实体定义详见[术语表 §2](./glossary.md#2-核心业务实体)，角色定义详见[术语表 §1](./glossary.md#1-角色与权限)。

## 2. P0：当前版本应保持稳定的能力

P0 是当前版本的基本盘（定义见 [PRD §3](./03-product-requirements-document.md#3-产品目标)），任何改动都不能破坏。

| 模块 | 能力 | 验证方式 |
|------|------|----------|
| 登录认证 | 三角色登录、JWT、路由守卫 | E2E 登录和越权测试（见 [PRD §11.1](./03-product-requirements-document.md#111-功能验收)） |
| 基础数据 | 电影、影院、分类、地区、演员、视频、公告 | 管理端页面和 CRUD |
| 排片 | 影院、影厅、电影、时间、价格、状态 | 影院端和管理端排片页 |
| 购票 | 电影详情、选择影院、选座、下单 | 用户端购票流程（见 [PRD §6.1](./03-product-requirements-document.md#61-用户购票流程) 泳道图） |
| 订单 | 用户、影院、管理端均可查看相关订单 | 订单页面 |
| 数据一致性 | 外键和初始化脚本可执行 | MySQL init.sql |

## 3. P1：建议下一阶段优先优化

### 3.1 订单状态机

现状：订单主要围绕待取票、已取票、已取消（当前定义见 [PRD §8.1](./03-product-requirements-document.md#81-订单状态定义) 和 [术语表 §4](./glossary.md#4-订单状态)）。

建议新增：

- 待支付
- 已支付
- 已出票
- 已取票
- 已取消
- 退款中
- 已退款

收益：

- 更贴近真实票务业务。
- 为支付、退票、核销打基础。

### 3.2 模拟支付

不一定立即接入真实支付，可以先做模拟支付：

- 下单后生成待支付订单。
- 用户点击“模拟支付”后变更为已支付。
- 超过一定时间未支付可取消。

收益：

- 保持项目可控。
- 能完整展示交易闭环。

### 3.3 取票码和核销

新增字段：

- `ticket_code`
- `paid_time`
- `picked_time`

新增流程：

- 用户支付后生成取票码。
- 影院端输入或点击核销订单。
- 订单状态变为已取票。

收益：

- 影院端价值更明显。
- 订单状态更真实。

### 3.4 对象级权限

现状：已有角色级权限，但影院端还需要“只能操作本影院数据”的对象级校验。

建议：

- 后端根据 JWT 中的影院 ID 过滤影院端订单、影厅、排片。
- 影院端新增/修改排片时，后端强制绑定当前影院。
- 管理员不受该限制。

收益：

- 安全性显著提升。
- 更符合真实 SaaS 多租户逻辑。

## 4. P2：商业化增强能力

### 4.1 座位模板

现状：用户端固定 8x8。

建议：

- 新增 `seat_template` 或在 `room` 中保存座位 JSON。
- 支持座位状态：可售、已售、不可售、过道、维修。
- 支持座位类型：普通座、情侣座、VIP 座。

收益：

- 可适配不同影厅。
- 为差异化票价打基础。

### 4.2 票价策略

建议支持：

- 按场次定价。
- 按座位区域加价。
- 按时间段优惠。
- 会员折扣。

收益：

- 影院端运营能力增强。
- 可增加收入分析维度。

### 4.3 优惠券与活动

建议支持：

- 满减券。
- 折扣券。
- 新用户券。
- 影院活动。

收益：

- 用户增长能力增强。
- 可做运营后台展示。

### 4.4 评价审核

现状：评价能力较基础。

建议：

- 评分和评论内容拆分。
- 增加评价审核状态。
- 增加敏感词或人工审核。
- 电影详情展示精选评价。

收益：

- 内容质量更可控。
- 平台治理能力增强。

## 5. P3：长期演进方向

| 方向 | 内容 |
|------|------|
| 多城市和定位 | 根据城市、距离、商圈筛选影院 |
| 个性化推荐 | 基于历史订单、评分、类型偏好推荐电影 |
| 会员体系 | 积分、等级、权益、储值 |
| 财务结算 | 影院收入、平台分账、票房统计 |
| 消息通知 | 邮件、短信、站内信、订单提醒 |
| 移动端适配 | H5/小程序/App |
| 开放平台 | 给影院或第三方接入排片和订单接口 |

## 6. 推荐版本规划

### V1.0 当前稳定版

- 保持当前三端闭环（详见 [PRD §4](./03-product-requirements-document.md#4-角色与权限) 角色定义和 [PRD §7](./03-product-requirements-document.md#7-功能需求) 功能清单）。
- 保证 E2E 59/59 通过。
- 文档、代码、数据库保持一致（见 [README 三方一致原则](./README.md#三方一致原则)）。
- 核心实体关系见 [PRD §8.2](./03-product-requirements-document.md#82-实体关系图-erd) ERD。

### V1.1 订单真实化

- 订单状态机。
- 模拟支付。
- 取票码。
- 影院端核销。
- 对象级权限。

### V1.2 影院运营增强

- 座位模板。
- 上座率统计。
- 场次收入统计。
- 票价策略。

### V2.0 商业化扩展

- 优惠券。
- 会员积分。
- 推荐系统。
- 真实支付。
- 退票改签。

## 7. 需求优先级建议

下一步最推荐先做：

1. 对象级权限：风险低、价值高，能让系统更像真实多租户后台。
2. 订单状态机：改动适中，是支付、取票、退票的基础。
3. 模拟支付 + 取票码：展示效果强，适合答辩和验收。
4. 座位模板：业务价值高，但涉及前后端和数据库联动，适合单独迭代。

## 8. PRD 优化检查表

每新增一个需求，至少补齐：

- 用户故事：谁在什么场景下需要什么能力。
- 业务规则：状态、权限、边界、异常。
- 页面变更：新增或修改哪些页面。
- 接口变更：新增或修改哪些 API。
- 数据库变更：新增字段、表、索引、外键。
- 验收标准：如何证明功能可用。
- 测试覆盖：是否需要 E2E 或单元测试。

> 以上标准对应的术语定义可查阅[术语表](./glossary.md)，当前功能状态可查阅 [PRD §7](./03-product-requirements-document.md#7-功能需求)。
